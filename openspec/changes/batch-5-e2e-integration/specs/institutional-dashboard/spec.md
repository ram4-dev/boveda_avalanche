# Delta for Institutional Dashboard

## ADDED Requirements

### Requirement: Batch 5 Runtime Mode And Evidence Visibility

The institutional dashboard MUST show the active Batch 5 runtime mode and MUST distinguish live Fuji-backed evidence from simulated demo evidence for portfolio, audit trail, loan detail, payment, and liquidation contexts.

#### Scenario: Dashboard labels Fuji-backed mode

- GIVEN the dashboard is opened through the normal `/` route with Fuji prerequisites supplied externally
- WHEN portfolio, loan detail, audit trail, payment, or liquidation evidence is displayed
- THEN the dashboard MUST identify the active mode as Fuji-backed or live mode
- AND live evidence SHOULD include tx hash, block number, contract address, receipt, or event source when available
- AND safe explorer links SHOULD be rendered only when a deterministic public explorer URL can be built from non-secret evidence.

#### Scenario: Dashboard labels demo mode

- GIVEN the dashboard is opened through `/demo`
- WHEN portfolio, loan detail, audit trail, payment, or liquidation evidence is displayed
- THEN the dashboard MUST identify the active mode as mock/demo mode
- AND tx hashes, events, receipts, or liquidation markers generated by mock state MUST be labeled as simulated
- AND the dashboard MUST NOT present simulated evidence as live Fuji finality.

### Requirement: Batch 5 Payment Attestation Dashboard Evidence

The institutional dashboard MUST reflect Batch 5 payment attestation outcomes with the same loan identity and evidence source used by borrower-facing payment surfaces and canonical event feed records.

#### Scenario: Dashboard reflects payment attestation evidence

- GIVEN an active loan receives a payment attestation in `/` or `/demo`
- WHEN the dashboard summary, portfolio, loan detail, or audit trail renders that loan
- THEN the dashboard MUST show payment status and attestation evidence such as attestation hash, transaction hash, receipt, event, or simulated marker when available
- AND it MUST show the same loan ID and compatible payment amount/currency as the borrower-facing surface and event feed
- AND missing live chain evidence MUST be labeled as pending, unavailable, or simulated instead of fabricated.

### Requirement: Batch 5 Liquidation Dashboard Evidence

The institutional dashboard MUST reflect Batch 5 liquidation outcomes with visible trigger context, final or pending status, USDC proceeds, distribution details when available, and compatible event-feed evidence.

#### Scenario: Dashboard reflects liquidation trigger and outcome

- GIVEN a loan enters margin-call, default, or liquidation flow
- WHEN the dashboard renders the affected loan, risk section, loan detail, or audit trail
- THEN it MUST show the trigger context such as current LTV, threshold breach, missed/defaulted payment, or canonical status
- AND it MUST show liquidation outcome and status
- AND proceeds currency MUST be displayed as `USDC`
- AND event evidence MUST be labeled as live Fuji, pending/unavailable, or simulated according to the active mode and returned evidence.

### Requirement: Batch 5 Explorer Link Integrity

The institutional dashboard MUST render explorer links only for live evidence items that include a real transaction, block, event, or contract reference and a safe deterministic public explorer target. The dashboard MUST NOT invent explorer links for missing or simulated evidence.

#### Scenario: Explorer link is rendered for live evidence

- GIVEN an audit event, loan detail, payment attestation, receipt, or liquidation record includes a live Fuji transaction hash, block number, or contract address
- WHEN the dashboard renders the evidence item
- THEN it SHOULD show a safe explorer link for that evidence
- AND the link label MUST make clear which tx, block, event, receipt, or contract evidence it represents.

#### Scenario: Explorer link is not rendered for simulated evidence

- GIVEN an evidence item comes from `/demo` or a mock adapter
- WHEN the dashboard renders that evidence
- THEN it MUST label the item as simulated or demo evidence
- AND it MUST NOT link the simulated marker as if it were live Fuji explorer evidence.
